## Project Overview

Personal wine cellar management project. Maintains a multi-year wine drinking plan synchronized with a live CellarTracker inventory and a Google Calendar menu plan. A nightly pipeline fetches data, generates food-wine pairing suggestions ("The Sommelier"), and publishes the plan as a web app.

## Architecture: Data / Presentation / Style

The site follows a clean separation:

- **Data** — `site/plan.json` (the plan), `site/pairing_suggestions.json` (pairings), `site/report.json` (inventory diff). All JSON.
- **Presentation** — `site/index.html` is an app shell with JS rendering logic. Fetches JSON data on load via `Promise.all`. No embedded data.
- **Style** — `site/style.css` contains all CSS. Sandstone color scheme (Bootswatch).

The app shell fetches `plan.json` and `pairing_suggestions.json` in parallel on load. If pairings are unavailable (404), the page renders without them.

## Running the Pipeline

```bash
bash fetch.sh
```

The pipeline (defined in `pipeline.sh`):
1. Resolves CellarTracker credentials from 1Password
2. Fetches inventory + menu calendar in parallel
3. Parses inventory + menu in parallel
4. Compares inventory vs `site/plan.json`
5. Generates pairing suggestions using inventory + menu + plan
6. Validates plan badges against CellarTracker Type field (auto-corrects `plan.json`)
7. Copies `pairing_suggestions.json` and `report.json` to `site/`

## LLM vs Data Boundary

- **CellarTracker is the system of record** for all wine metadata: Type/badge, Varietal, Appellation, scores, drinking windows, quantity.
- **The LLM provides:** scheduling decisions (which bottle in which week, seasonal fit, urgency ordering) and plan generation (Step 3 in fetch.md).
- The LLM must never invent or override wine metadata. `validate_plan.py` auto-corrects any badge mismatches after plan generation.

## Architecture of site/index.html

- Pure vanilla HTML/CSS/JS — no build step, no dependencies, no framework.
- Fonts loaded from Google Fonts (Playfair Display, DM Sans).
- SVG wine glass icons distinguish wine types (champagne flute for sparkling, varying fill levels for rosé/white/red).
- Five tabs: Year 1, Year 2, All 104 Weeks, Monthly Calendar, What Changed.
- Weeks begin on Monday.
- Legend explains wine type icons, tile border colors (urgent/special/evolution), and pairing indicators (pairs well / sommelier pick / no match).

## Google Calendar Integration

- Menu data comes from a Paprika-managed calendar via a secret .ics URL.
- The URL is stored in `.env` as `GOOGLE_CALENDAR_ICS_URL`.
- `parse_menu.py` converts UTC timestamps to Pacific time (`America/Los_Angeles`) and includes meals from the current week's Monday through a 2-year horizon.
- Keywords are extracted using the canonical list in `wine_keywords.py`.

## Wine-Food Pairing Engine ("The Sommelier")

- `wine_keywords.py` is the single source of truth for food keywords and their pairing rules, used by both `parse_menu.py` (extraction) and `pairing.py` (scoring).
- Three pairing outcomes displayed on cards:
  - **Pairs Well** (green) — planned wine matches the meal
  - **Sommelier Pick** (blue) — suggests a specific bottle with window and score
  - **No Match** (muted) — no food keywords matched, enjoy the planned wine
- Bottle suggestions use the priority rules: past peak → expiring this year → expiring next year → peak window.
- Only bottles in their drinking window (BeginConsume ≤ current year + 1) are suggested.
- Each meal gets a unique suggestion — the engine avoids recommending the same bottle twice.

## CellarTracker Integration

- Export API endpoint: `https://www.cellartracker.com/xlquery.asp` with query params `User`, `Password`, `Table=Inventory`, `Format=tab`, `Location=1`.
- Returns tab-delimited text (one row per physical bottle). Key fields: Vintage, Wine, Varietal, Type, Country, Region, SubRegion, Appellation, iWine, Quantity, Location, Bin, Size, BeginConsume, EndConsume, CT, MY.
- **Credentials must never be written to any file.** Resolved at runtime from 1Password via `op read`.

## Important Conventions

- When matching inventory bottles to plan entries, match on vintage year + wine name (fuzzy, with alias expansion — see `compare.py` ALIASES dict).
- Drinking window urgency: bottles past `EndConsume` or within 1 year of it are considered urgent.
- Plan updates: edit `site/plan.json`, preserve the `allWeeks` format, update the `changelog` object.
- Pairing scores are string constants: `"good"`, `"partial"`, `"poor"`, `"neutral"` — used across Python and JS.
- `pipeline.sh` contains all shared pipeline logic. `fetch.sh` and `fetch_docker.sh` are thin wrappers.
